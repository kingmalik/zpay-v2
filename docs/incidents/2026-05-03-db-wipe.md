# Incident: Postgres Wipe — 2026-05-03 ~09:25 AM PDT

## Summary

Railway Postgres for `zpay-v2-production` was reprovisioned at approximately 09:25 AM PDT on 2026-05-03.
All W1–W14 production data (7,067 rides, 226 drivers, 26 batches, driver balances, rates) was lost.
Recovery took ~6 hours from local CSVs and yielded ~95% of data with permanent gaps.

## Root Cause (most likely)

Railway logs for this time window are not retained (Railway keeps ~hours, not days, of container logs).
The exact trigger was NOT recoverable from logs at time of investigation (2026-05-03 23:30 PDT).

**Most plausible cause:** Railway's Postgres service was reset/reprovisioned — either:
a) An automated Railway-internal maintenance action on the Postgres volume
b) A manual action in the Railway dashboard (unlikely — Malik was asleep)
c) Triggered by the Alembic orphan-head crash loop from May 2 evening, where repeated failed
   deploys may have caused Railway to reset the service state

The May 2 evening incident (PR #47/#48 fixed a 3-way Alembic head conflict) is the most
suspicious precursor. The deploy storm from orphan migrations may have caused Railway to
reprovision the Postgres service after repeated crash-loop restarts.

## Contributing Factors

1. **No off-host backups**: The only backup was `/data/out/zpay_backup.sql` on Railway disk —
   which gets wiped when Postgres is reprovisioned.

2. **Silent auto-restore**: The existing auto-restore in `entrypoint.sh` ran but produced no alert.
   Backend started normally with 0 rows — no Discord notification, no crash.

3. **No boot guard**: Nothing checked whether the DB had data after startup. The wipe was
   invisible to all monitoring for ~6 hours.

4. **No external monitoring of row counts**: Health monitor checked DB liveness (SELECT 1) but
   not data presence.

## Fixes Applied (2026-05-03 overnight recovery session)

- Phase 1A: Manual encrypted SQL snapshot via Railway TCP proxy → `~/Library/Application Support/zpay-backups/manual/2026-05-03_post-recovery.sql.gpg`
- Phase 1B: Mac launchd watcher auto-vaults mom's weekly Excel/CSV files within 60s
- Phase 1C: Committed SQL dumps removed from git working tree (`3e229f7`)
- Phase 2: Hourly backup pipeline + daily CSV export added to APScheduler (gated by `BACKUP_CRON_ENABLED`)
- Phase 2C: `scripts/restore_from_vault.py` one-command restore with sanity checks
- Phase 3B: Boot guard added to `app.py` lifespan startup — exits non-zero if ride count is 0 and `ALLOW_EMPTY_DB != 1`
- Bug fix: `driver_scorecard.py` was referencing `r.trip_date` (column doesn't exist) — fixed to `DATE(r.ride_start_ts)`, was flooding Postgres error logs every ~10 seconds
