## Broken / Risky Business Logic

### 1. Pay stub total does not match payroll summary total
- **Impact:** Drivers may receive a pay stub showing a different dollar amount than what payroll calculates.
- **Where:** `email.py` and `rides.py` both generate PDFs independently. `email.py::_generate_pdf` loops over raw rides and computes `net = z_rate - deduction`, accumulating `printed_total`. The payroll summary (`summary.py::_build_summary`) adds `from_last_period` (carried-over balance) to `net_pay`. The PDF stub never includes the carry-forward amount — so a driver who earned $80 last week + $60 this week ($140 total, cleared) will receive a stub showing only $60.
- **Fix:** Pass `from_last_period` into `_generate_pdf` and print it as a line item. Or generate the PDF from the same `_build_summary` rows.

### 2. `auto_save=True` runs on every page load of `/summary`
- **Impact:** Each time anyone loads the summary page (including Malik browsing), the system silently writes or deletes `DriverBalance` records. If mom loads the page mid-week before all rides are uploaded, it will write incorrect carry-forward values that persist into the next payroll cycle.
- **Where:** `summary.py` line 217: `_build_summary(..., auto_save=True)`. There is no confirmation step.
- **Fix:** Only auto-save on an explicit "Run Payroll" / "Lock Batch" button press. Add a separate POST endpoint for saving; keep GET as read-only.

### 3. Carry-forward logic uses `period_start` for batch ordering, not upload order
- **Impact:** If a batch is uploaded out of order (e.g., Week 8 before Week 7), the "previous batch" lookup picks the wrong batch and pulls the wrong carry-forward amounts, potentially double-paying or erasing a withheld balance.
- **Where:** `summary.py` lines 108-112 — `filter(period_start < current_batch.period_start)`. No lock on batch completeness.
- **Fix:** Add a "batch locked" flag to `PayrollBatch`. Only count locked batches as prior batches for carry-forward lookups.

### 4. `rides.py::rides_update` sets `gross_pay = z_rate`
- **Impact:** When a ride rate is manually edited, gross_pay is overwritten with the z_rate. For Acumen rides, gross_pay represents what FirstAlt paid; z_rate is what the driver costs. Overwriting gross_pay destroys the profitability calculation for that ride.
- **Where:** `rides.py` line 267: `ride.gross_pay = z_rate`.
- **Fix:** Do not touch `gross_pay` in the manual rate update path. Only update `z_rate`, `deduction`, and `net_pay`.

### 5. `send-all` silently swallows individual email failures
- **Impact:** If 3 of 30 email sends fail (bad address, SMTP timeout), mom sees `skipped=3` with no way to know which 3 drivers were missed.
- **Where:** `email.py` lines 253-263: bare `except Exception: skipped += 1` with no driver name capture.
- **Fix:** Collect failed driver names; display them on the redirect target page.

### 6. MAZ upload creates a second DB session, discarding the injected one
- **Impact:** The MAZ upload endpoint accepts `db: Session = Depends(get_db)` but discards it and opens a raw `SessionLocal()` at line 215. The injected session transaction is abandoned. Any state set before the insert is lost.
- **Where:** `upload.py` line 215: `db: Session = SessionLocal()`.
- **Fix:** Remove the `SessionLocal()` call and pass the injected `db` into `bulk_insert_rides`.

### 7. Debug code left active in production (MAZ upload)
- **Impact:** Every MAZ upload runs a targeted "wanted keys" debug loop that prints to stdout. Performance drain and leaks internal ride key patterns.
- **Where:** `upload.py` lines 218-234.
- **Fix:** Remove the `wanted` / `hits` debug block.

### 8. `rates_set` permanent update has no source filter — affects both companies
- **Impact:** If a service name exists in both Acumen and Maz (possible with generic names), setting a permanent rate from the unmatched UI overwrites z_rate for rides from both companies.
- **Where:** `rates.py` line 185: `db.query(Ride).filter(Ride.service_name == service_name)` with no `Ride.source` filter.
- **Fix:** Add `Ride.source` filter when scoping the permanent rate update.

### 9. `validate.py` hardcodes paths to Malik's Mac
- **Impact:** The validator only works on Malik's machine. Silently shows no results on any other machine (including mom's) without an error.
- **Where:** `validate.py` lines 26-27: `/Users/malikmilion/Downloads/Acumen` and `/Users/malikmilion/Downloads/Maz`.
- **Fix:** Make paths configurable via environment variables, or support file upload directly in the UI.

### 10. `ZRate` model imported in `rides.py` but does not exist
- **Impact:** The `rides.py::update_rate` POST endpoint references `ZRate` which is not a model in `models.py`. Will raise a `NameError` if triggered.
- **Where:** `rides.py` line 344.
- **Fix:** Remove the dead endpoint or use the correct model (`ZRateService` / `ZRateOverride`).

### 11. Deduction not subtracted in payroll summary `net_pay` sum
- **Impact:** `_build_summary` sums `Ride.z_rate` as the driver's net pay, but ignores `Ride.deduction`. If any ride has a deduction applied, the driver is overpaid relative to what the pay stub shows.
- **Where:** `summary.py` line 78: `func.coalesce(func.sum(Ride.z_rate), 0).label("net_pay")`.
- **Fix:** Use `func.sum(Ride.z_rate - Ride.deduction)` or ensure deductions are baked into z_rate before summarizing.

## High-Value Missing Features

### 1. Driver onboarding form
- **Why it matters:** New drivers can only enter the system via file upload or FirstAlt sync. There is no way for mom to manually add a driver before their first ride file arrives.
- **Effort:** Low — simple `/people/new` form creating a `Person` record.

### 2. Driver status tracking (active / inactive / suspended / terminated)
- **Why it matters:** The `Person.active` boolean is too coarse. No way to mark a driver as suspended, on leave, or terminated — or surface that status on dispatch/payroll.
- **Effort:** Low — add a `status` enum field; surface on driver profile and dispatch cards.

### 3. Driver profile page (all fields in one place)
- **Why it matters:** `Person` stores phone, home_address, email, and both partner IDs, but they are never all shown together. Mom needs a single-screen view per driver.
- **Effort:** Low — `/people/{person_id}/profile` page with all fields editable.

### 4. Payroll lock / approval workflow
- **Why it matters:** There is no "lock batch" concept. `auto_save=True` on every GET means payroll is never finalized. No clear moment when results are authoritative and immutable.
- **Effort:** Medium — add `locked_at` and `locked_by` to `PayrollBatch`. Block auto_save and edits after lock. Add "Lock Payroll" button.

### 5. 1099 / annual earnings report
- **Why it matters:** All drivers are independent contractors requiring annual earnings summaries for tax filing. Z-Pay has the data but no annual view.
- **Effort:** Medium — `/reports/1099` grouped by driver and calendar year, exportable to Excel.

### 6. Payment history per driver (driver-first view)
- **Why it matters:** No page shows every payment ever made to a specific driver. The current drill-down is batch-first, requiring 4 clicks to find one driver's history. Driver disputes have no resolution tool.
- **Effort:** Medium — `/people/{person_id}/history` listing all batches a driver appeared in with amounts paid, withheld periods, and ride counts.

### 7. Batch completeness check before payroll
- **Why it matters:** If mom runs payroll on a partially-uploaded batch, drivers with zero rides that week will be underpaid. No current warning that known drivers are missing.
- **Effort:** Medium — compare current batch driver list against the previous batch for the same company; alert if any prior-batch driver has zero rides in the current batch.

### 8. Rate change audit log
- **Why it matters:** When a rate is edited (rates_set or admin/rates), there is no record of what it was before, who changed it, or why. Compliance and dispute risk.
- **Effort:** Low — add a `rate_change_log` table (service_id, old_rate, new_rate, changed_at, reason) and write on every update.

### 9. Weekly profit trend chart
- **Why it matters:** Analytics shows profitability tables but no time-series. Malik cannot see whether profit is trending up or down without mentally comparing rows. The `period_rows` data is already built.
- **Effort:** Low — add a Chart.js line chart to `intelligence.html` using the existing `period_rows` data.

### 10. Duplicate upload detection at the UI level
- **Why it matters:** DB-level deduplication works (via `source_ref`), but mom sees `inserted=0, skipped=N` which is confusing — not a clear "already uploaded" message.
- **Effort:** Low — check for existing batch with same `period_start` + `source` before insert; show prominent warning on success page.

### 11. Email delivery confirmation log
- **Why it matters:** No record of which drivers were emailed, when, or for which period. Cannot verify delivery if a driver disputes receiving their stub.
- **Effort:** Low — `email_log` table (person_id, batch_id, sent_at, to_email, status); write on every `send_paystub` call.

### 12. Driver availability tracking
- **Why it matters:** Smart-assign scores drivers by location and schedule overlap but has no concept of availability. Assigned drivers may be on a scheduled day off.
- **Effort:** High — new `driver_availability` table with date ranges and day-of-week patterns; UI to set and display.

## Data Collected But Never Used / Never Shown

### 1. `Person.phone` and `Person.home_address`
- Stored in the model. Never displayed in the people directory, dispatch cards, or any export. Useful on dispatch (call driver quickly) and for 1099 mailing address.

### 2. `Person.external_id`
- Shown in payroll summary as "Code" but never editable in the UI. Purpose unclear. Should be documented and made editable on the driver profile page.

### 3. `Ride.spiff`
- Ingested from Acumen files and stored. Never included in pay stub calculations or displayed on any analytics page. If spiff is bonus pay, drivers cannot see it and it does not appear in profitability calculations.

### 4. `Ride.deduction` (inconsistently applied)
- Present in the model and rendered in pay stub PDFs. But the payroll summary sums `z_rate` only — deductions are ignored. Creates a discrepancy between what the stub shows and what payroll calculates.

### 5. `ZRateOverride.reason`
- The reason field exists on rate overrides and is insertable via the admin form but is never displayed on the override list, analytics pages, or audit views.

### 6. `PayrollBatch.notes`
- Notes field exists in the model but is never shown in the batch list, history, or detail views, and there is no UI to set it.

### 7. `DispatchAssignment` records (completely orphaned)
- Assignments confirmed via smart-assign are written to `dispatch_assignment` but are never read back anywhere. No page lists today's or historical confirmed assignments. No way to cancel.

### 8. `Ride.z_rate_source`
- Tracks whether the rate came from an override, service default, or one-time entry. Only visible in the raw `/rides/data` JSON endpoint, not on any payroll or analytics view.

### 9. `Ride.source` not shown in per-driver ride views
- The source (acumen vs maz) is on every ride but not shown on the per-driver ride detail page. Drivers working both partners in one batch cannot tell which rides came from which partner.

### 10. `Ride.gross_pay` vs `Ride.net_pay` — relationship unexplained
- Both fields are stored and used in profitability calculations but the semantic difference (gross = what partner pays; net = after partner deductions; z_rate = what driver costs) is never explained in the UI. This creates confusion about what "profit" means.

## Automation Opportunities

### 1. Scheduled payroll email send
- **Manual process:** Mom must remember to click "Send All" after every payroll run for every company.
- **How to automate:** Trigger `send-all` automatically after a batch is locked, or via a configured schedule (e.g., every Friday at 5pm).

### 2. Nightly DB snapshot
- **Manual process:** The snapshot/save endpoint exists but must be manually triggered from the UI.
- **How to automate:** Run pg_dump nightly via APScheduler. Store multiple timestamped backups, not just one overwritten file.

### 3. Auto-recalculate rates after rate changes
- **Manual process:** After editing a rate in admin, user must click "Recalculate" to update existing rides.
- **How to automate:** Hook recalculate into `rates_set` and `set_default_rate` as a background task automatically.

### 4. Partner file monitoring (email polling)
- **Manual process:** Mom must download files from email and upload them to Z-Pay manually.
- **How to automate:** IMAP watcher monitoring Acumen/Maz inboxes, auto-downloading attachments to the upload directory, then triggering the importer. High value, medium-high effort.

### 5. Zero-rate alert on upload completion (Maz parity)
- **Manual process:** After uploading, user must navigate to Alerts to check unmatched rates.
- **How to automate:** Show the unmatched count on the upload success page. Already done for Acumen; not done for Maz (which silently redirects to `/summary`).

## Mom's Workflow Gaps

### 1. No confirmation before payroll balances are written
- Mom loads `/summary` and `auto_save=True` immediately writes carry-forward balances. No "are you sure?" step. Accidental navigation could corrupt the next week's payroll.

### 2. Maz upload redirects to `/summary` with no feedback
- After uploading a Maz PDF, mom is silently sent to the summary page with no ride count, skip count, or rate mismatch warning. Compare to Acumen which shows a proper success page with all this information.

### 3. No driver email roster check before bulk send
- Before sending, there is no warning that X drivers have no email address on file. Mom will click "Send All," get back `skipped=5`, and have no idea which 5 drivers to follow up with.

### 4. People navigation requires 4-5 clicks to reach a driver's rides
- To see a driver's rides: pick company → pick batch → pick week → pick driver → rides. A driver name search that jumps directly to their current-week rides would be far simpler for a non-technical user.

### 5. No visual distinction between Acumen and Maz batches
- Payroll history and batch list show company names in text but no color coding. Mom must read carefully to avoid confusing the two companies.

### 6. Snapshot/backup is invisible to mom's workflow
- The snapshot feature exists but is only accessible if mom knows the URL. There is no "save backup before running payroll" prompt or checklist.

### 7. Alerts badge always shows high count due to "inactive drivers"
- Any driver with no rides in 60 days is flagged as inactive. In a seasonal or on-call business, this could be 30+ drivers at any time. Mom will see a large badge and have no way to dismiss false positives, causing alert fatigue.

## Quick Wins
_(Low effort, high immediate value)_

1. **Maz upload success page** — Wire Maz upload to show `upload_success.html` instead of silently redirecting. Already has `inserted`/`skipped`/`by_person` data in scope before the redirect.

2. **Email pre-flight check** — Before "Send All," show a count of "X have email / Y missing email" and list the missing drivers by name. One template query, no backend changes.

3. **Show `z_rate_source` as badge on ride rows** — Add a small colored badge ("override" / "default" / "one-time") next to each rate in `people_person_rides.html`. Data already in scope.

4. **Sum total withheld on payroll summary** — The `totals` dict in `_build_summary` omits the total withheld amount. Add `sum(withheld_amount for row in rows if row['withheld'])` — one line.

5. **Warn on empty batch** — If `total_rides == 0` when batch_id is selected on summary, show a red banner "This batch has no rides uploaded yet." Prevents payroll on empty batches.

6. **Show `spiff` on pay stubs** — The `spiff` field is ingested and stored. Add it as a line item in `_generate_pdf`. Drivers currently have no visibility into bonus pay.

7. **Driver search on dispatch page** — Add a client-side text filter input on `dispatch.html`. One line of JavaScript to filter the driver card list by name.

8. **Rate change notes on admin overrides** — The `ZRateOverride.reason` field is stored but never displayed. Show it in the overrides list table. Template-only change.

9. **Fix dead `ZRate` import in rides.py** — `rides.py::update_rate` references a model that does not exist. Remove or fix before it causes a runtime crash in production.

10. **Add Ride.source badge on per-driver ride detail** — Show which partner (Acumen / EverDriven) each ride came from on the `people_person_rides.html` page. The `Ride.source` field is available; add a small colored tag.
